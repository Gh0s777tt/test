# Monitor Drift Escalation Policy

This policy defines escalation levels for repeated monitor drift in benchmark governance.

---

## 1) Purpose

The monitor stage is intentionally non-blocking, but repeated drift can indicate:

- unstable threshold policy,
- environment regressions,
- benchmark fidelity issues, or
- real performance regressions.

This policy standardizes when and how drift is escalated.

---

## 3) Escalation Levels

### Level 0: `normal`

Typical conditions:

- no repeated monitor drift pattern,
- no material monitor failure signal.

Expected action:

- continue routine monitoring.

### Level 1: `watch`

Typical conditions:

- sporadic drift in recent runs, or
- moderate drift rate trend.

Expected action:

- add explicit note in weekly governance review,
- keep recommendation/dashboard snapshots in evidence bundle.

### Level 2: `escalated`

Typical conditions:

- repeated drift streak across recent runs (for example, at least 2 consecutive drift results),
- elevated drift-rate trend,
- monitor failure-rate trend that is no longer incidental.

Expected action:

- open or update a MONPOL proposal in the next release cycle,
- assign owner and include mitigation plan,
- include latest escalation artifact in proposal evidence.

### Level 3: `critical`

Typical conditions:

- sustained drift streak (for example, at least 3 consecutive drift results), or
- repeated timeout/missing-report/failed monitor outcomes,
- severe drift/failure-rate trend.

Expected action:

- immediate triage in current release cycle,
- temporary containment decision (hold/relax/tighten with explicit rationale),
- incident-style follow-up in changelog and signoff records.

---

## 4) Default Advisory Thresholds

Default threshold set used by automation:

- watch drift rate: `20%`
- escalated drift rate: `40%`
- critical drift rate: `60%`
- escalated consecutive drift: `2`
- critical consecutive drift: `3`
- escalated failure rate: `10%`
- critical failure rate: `20%`

These thresholds are advisory defaults and may be calibrated with MONPOL governance.

## 8) Governance Requirements

When overall level is `escalated` or `critical`:

1. proposal evidence should include latest escalation artifact,
2. changelog rationale should mention escalation context,
3. reviewer signoff should confirm mitigation plan and follow-up.

Governance gate promotion strategy is defined in:

- `governance/performance/MONITOR_THRESHOLD_GOVERNANCE_GATE_PROMOTION.md`
- `governance/performance/MONITOR_THRESHOLD_GOVERNANCE_GATE_PROMOTION.json`
